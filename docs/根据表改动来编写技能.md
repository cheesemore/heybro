# 根据表改动来编写技能

本文约定：**当 `skills.json`、`wowBookMonsters.json`、`wowBookBosses.json` 或策划表发生变动，需要新增或修改技能时**，人机协作应遵循的流程。

**触发语（不必带路径）**：你在对话里说 **「更新技能」**（或同义短句，如「把 pending 的技能对齐」「把 missing 接上」），即视为要求执行方 **默认按本文执行**，不必再说「按 `docs/根据表改动来编写技能.md` …」整句。若你还有**额外范围**（例如只处理某几条 id、或结合某次表 diff），在「更新技能」后补上即可。

---

## 1. 范围说明

| 层级 | 内容 |
|------|------|
| **配置** | `src/game/config/skills.json`：id、nameCn、limit、powerTag、codeStatus、descriptionCn、logicEffectCn、params 等。 |
| **挂载** | 小怪 `skillIds`、书本首领 `skills` 等，须与 `skills.json` 中 id 一致。 |
| **代码** | `BattleScreen`、专用 `cast*` 等战斗逻辑；未接战斗的 `codeStatus: missing` 可只改表不写代码。 |

「编写技能」可能仅改 JSON，也可能要改 TypeScript；以每条技能的**实际缺口**为准。

---

## 2. 总流程（执行方可自行拆分步骤，但不得跳过「确认」节）

### 阶段 A — 先遍历分类（默认第一步，不可省略）

你说「更新技能」且未另指定范围时，执行方应 **先读 `src/game/config/skills.json`**，自动整理成下面几类（用表格或列表输出 **skill id + nameCn + codeStatus（+ 若有 powerTag）**），**再**进入逐条确认（阶段 B）：

| 分类 | 建议判定（以表为准，可补充说明） | 执行方意图 |
|------|----------------------------------|------------|
| **待改动（优先）** | `codeStatus === "pending_changes"` | 图鉴编辑器里已保存、标记为「待修改」——通常表示 **表/文案已动，战斗或程序待对齐**；必须逐条先读后改、等你确认。 |
| **需新增 / 未接战斗** | `codeStatus === "missing"`（或你点名的新 id） | 表里已有条目但战斗侧未接、或新登记技能；区分「只改 JSON」与「要写 TS」后再走 B。 |
| **已写** | `codeStatus === "written"`（或 `pending_changes` 以外的已接状态） | 默认 **本轮不自动改**，除非你同时提供了表 diff、或你明确列出要动的 id。 |

此外可按需补充（与本次是否相关由你一句话说明即可）：

- **挂载引用**：在 `wowBookMonsters.json` / `wowBookBosses.json` 等里出现的 `skillIds` / `skills`，是否在 `skills.json` 都有定义（缺 id 的单独列「缺表」）。
- **其它依据**：策划表 diff、PR 描述、编辑器未保存文件等——若你提供了，再在 A 阶段合并进「待处理列表」，并标注来源。

**阶段 A 的输出物**：至少包含两段清单——**「本轮建议作为『改动』处理的 id」**、**「本轮建议作为『新增/接战斗』处理的 id」**；若有歧义（例如 `written` 但你声称表已改），单独标「需你一句话定范围」。

### 阶段 B — 逐条技能处理（核心）

对列表中的 **每一个** skill id（含新增），必须单独走一遍下面子流程 **B1→B4**，**禁止**在未获确认前批量改文件。

#### B1 读取现状

- 在 `skills.json` 中查找该 id；**若已存在**，把该条 **完整 JSON（或等价结构化摘要）** 原文读出备用。
- 若代码中有引用（`getSkillById`、`skillParamNumber`、专用 cast 等），执行方应自行检索并摘要，不必一次贴完全部代码，但**修改意图依赖的调用点**要说清。

#### B2 拟定方案（执行方输出）

用固定结构汇报（便于你扫读）：

- **id**：
- **类型**：新增 / 仅配置 / 需改战斗逻辑
- **当前摘要**：（已有则贴原表关键字段；无则写「无，拟新增」）
- **拟修改**：逐条列出将改动的字段或代码文件、行为变化（玩家可见效果 + 数值/参数）
- **风险**：如影响 limit、powerTag、与其它技能的交互等

#### B3 向你提问并等待确认

- **必须**有一句明确提问，例如：「以上方案是否按此实施？请回复确认 / 调整 xxx 后再做。」
- **修改已有技能时**：必须先完成 **B1 原样展示 + B2 拟改说明**，再提问；不得省略原文关键字段。
- 你回复「确认」「OK」「按 B2 做」等视为该条通过；若你调整方案，执行方应 **更新 B2 后再问一次**，直到该条明确通过。

#### B4 实施与自检

- 仅在 **该条 skill 已获得你确认** 后，才写入 `skills.json` 或改 TS。
- 自检：`skills.json` 能通过现有校验（如 `skillsCatalog.ts` 加载、`params` 长度等）；若改了代码，说明如何本地验证（如进战斗、单测）。

### 阶段 C — 收尾

1. 汇总本轮已确认并落地的 skill id 列表。
2. 若有未确认项，单独列为「待你确认」，不混入已完工描述。

---

## 3. 你如何发起（简短即可）

- **默认**：「**更新技能**」→ 执行方按本文：先做 **阶段 A 遍历分类**，再对列出的每条做 **B1→B3** 确认，最后 **B4**。
- **带范围**：「更新技能，只处理 `skill_xxx`」「更新技能，结合下面表 diff：…」→ 在 A 阶段把你的范围与 `skills.json` 扫描结果 **合并去重** 后再走 B。

执行方在收到「更新技能」后应：

1. **先做阶段 A**（遍历：待改动 / 新增或 missing / 可选挂载检查）；  
2. 再对 **每条** 待处理 skill 执行 **B1→B3**，**每条一次确认**；  
3. 全部确认后再集中或逐条做 **B4**（不得跳过确认）。

---

## 4. 执行方禁止事项（与你的约定对齐）

- 不得在未展示**原技能**（若存在）且未说明**拟改方案**的情况下，直接修改已有技能条目。
- 不得把多条技能的确认合并成一句「以上全部确认吗」**代替**逐条展示与逐条确认（除非你明确声明本轮采用批量确认，且仍须逐条列出 B1/B2）。
- 不得假设你已默认同意「和表一致即可」而省略 **B3 提问**。
